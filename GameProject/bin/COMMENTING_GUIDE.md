# Commenting Guide (설명 주석 가이드)

이 프로젝트는 **"무엇을 하는 코드인지"보다 "왜 이렇게 했는지/어디를 고치면 되는지"**가 유지보수에 더 중요합니다.
따라서 아래 규칙으로 **설명 주석(maintenance comments)** 을 작성합니다.

## 목표
- 신규 투입자가 파일을 열었을 때 30초 안에 흐름 파악
- 수정 시 어디를 건드려야 하는지(변경 포인트) 빠르게 찾기
- 주석이 코드와 함께 오래 살아남도록(거짓말 주석 방지)

## 하지 말아야 할 주석(소음)
- `i++ // i를 1 증가` 같은 자명한 설명
- 코드와 바로 불일치하게 되는 값(예: 하드코딩 값 나열)

## Java 파일 권장 템플릿
### 1) 클래스 상단(JavaDoc)
- 역할/책임
- 입력/출력(요청 파라미터, 반환)
- 주요 에러/예외
- 변경 포인트

예시:
```java
/**
 * [역할] 회원 관련 MVC2 컨트롤러
 * [URL] /member/*
 * [변경 포인트] join/login 파라미터가 바뀌면 여기와 JSP를 같이 수정
 */
```

### 2) 메서드 상단(짧은 블록 주석)
- 이 메서드가 호출되는 시점
- 검증/실패 시 경로

### 3) "왜" 주석
- 보안/우회 방지
- 성능
- 제약(외부 시스템, DB 스키마)

## JSP 파일 권장 템플릿
- 페이지 목적
- 필요한 request/session attribute
- 폼 action / 기대 파라미터
- 유지보수 포인트(라우팅, 파라미터)

## Controller/Service/DAO 책임 분리 주석
- Controller: 라우팅/입력 수집/뷰 이동
- Service: 비즈니스 규칙(검증, 정책)
- DAO: DB 접근만(쿼리/트랜잭션 경계 주의)

## 주석 업데이트 원칙
- 로직을 바꾸면 주석도 같이 바꾸기
- 주석이 의심스러우면 삭제하고 다시 쓰기(낡은 주석이 최악)
